SS 的故障修复一例

发表于 2016-07-16 |

一直在使用 ShadowSocks 这把梯子，本身是极为不错的，平时极少拉稀摆带，关键时候从没认过怂。半年左右时间，一直相安无事。故事从大约两三个月前开始，忽然，梯子不好用，墙外的世界再度被隔离。后来在 dongua 推荐下试用蓝灯，世界敞亮大约一个多月，及至最近两周之前，蓝灯也开始不好用，从此再无梯子，搜索只能用回 sogou。

最初以为是 SS 协议和蓝灯的协议被功夫网封死，因为试过几次切换 SS 的源，问题仍然存在。还是前天受老寇启发，他说他这么长时间都用 SS 是可以的，没有出现过问题，我一下子才想到可能是自己的环境有问题。

最终经过排查，发现是 chrome 的插件『Host Switch Plus 1.3.9』有问题。在 chrome 的扩展管理里面禁用掉它之后，就能恢复梯子。而这是一个可以在 chrome 下面创建和切换 host 环境的插件。遥想当年 FireFox 的 Host 插件，真是让人怀念，所以当时一看到这个插件，我就装上了。没想到出了这么一个幺蛾子。

而在其功能界面，也明确提供了针对修改浏览器代理模式的功能。

众所周知，无论是 SS 还是蓝灯，抑或 SwitchyOmega 扩展等第三方工具，都是利用系统 PAC 机制实现科学上网功能的。HSP 修改 PAC 的时候，其实等于覆盖了 SS 及蓝灯的设置，当然导致无法再上网。并且，这种修改是覆盖式的，HSP 没有『恢复设置』一说，也就是说，一旦修改之后，无论你在 HSP 的界面将上网模式设置成『DIRECT』还是『SYSTEM』都不能正确翻墙。除非你在 chrome 的扩展管理界面中将 HSP 禁用或者卸载，让 chrome 帮你恢复系统的 PAC 设置。

PS：关于 SS 不能使用的问题，网上还有网友提到是因为迅雷的一个 DLL 文件作祟，以及迅雷默认端口与 SS 默认端口冲突所致。Windows 下的朋友遇到的话，也可以检查一下这种情况。
